Unfortunately, but true all the same, some legacy systems are so counter to the DDD way of
designing with
Bounded Contexts
that you might even refer to them as
unbounded
legacy systems.
That’s because such a legacy system is what I’ve already referred to as a
Big Ball of Mud.
In reality
the one system is full of multiple tangled models that should have been separately designed and
implemented but were jumbled together into one very complex and intertwined mess.
Stated another way, when we are discussing a legacy system there are probably some, even many,
logical
domain models that exist inside that one legacy system. Think of each of those logical domain
models as a
Subdomain.
In the diagram, each logical
Subdomain
in the unbounded legacy monolithic
Big Ball of Mud
is marked off by a dashed box. There are five logical models or
Subdomains.
Treating the logical
Subdomains
as such helps us grapple with the complexity of large systems. This
makes a lot of sense because it allows us to treat the
problem space
as if it had been developed using
DDD and multiple
Bounded Contexts.
